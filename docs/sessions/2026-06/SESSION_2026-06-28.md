## 📝 Décisions Techniques
- Utilisation de `SUPABASE_ANON_KEY` pour l'écriture ET la lecture (secrets déjà configurés)
- Policy INSERT restreinte à `source = 'github-actions'` pour sécuriser l'accès public
- Génération d'un UUID côté workflow Linux (`/proc/sys/kernel/random/uuid`) car `cuid()` est côté client Prisma
- Pas de migration Prisma classique à cause d'une ancienne migration `init` corrompue — `prisma db push` utilisé à la place

---

## 🔍 Découvertes
- Le projet Supabase était en pause malgré 143 runs du workflow (dont beaucoup de runs manuels)
- L'API REST retourne HTTP 400 si `id` n'est pas fourni car `cuid()` n'est pas généré côté DB
- Les noms de colonnes dans l'API REST Supabase utilisent les noms PostgreSQL (`checked_at`), pas les noms Prisma (`checkedAt`)
